Einleitung

Die vorliegende Dokumentation stellt eine umfassende Anleitung dar, wie Runit als Init-System und Prozess-Supervisor in Verbindung mit dem Apache HTTP Server auf einem FreeBSD-Betriebssystem eingesetzt werden kann. Runit ist ein leichtes, aber äußerst leistungsstarkes Dienstprogramm, das auf Unix-ähnlichen Betriebssystemen wie Linux und BSD verwendet wird. Es bietet automatische Überwachung und Dienstwiederherstellung im Falle eines Ausfalls und ermöglicht die sichere Verwaltung von Prozessen und Diensten, einschließlich des Apache-Webservers.
Der Apache-HTTP-Server ist einer der am weitesten verbreiteten Webserver der Welt und wird von vielen Websites zum Teilen von Inhalten im Web verwendet. Um einen Webserver wie Apache erfolgreich auf dem FreeBSD-Betriebssystem zu betreiben, ist es wichtig, dass er stabil und zuverlässig ist. Die Verwendung von Runit und Apache HTTP Server auf dem FreeBSD-Betriebssystem bietet die ultimative Lösung für eine leistungsstarke Webhosting-Umgebung.

Was ist FreeBSD

FreeBSD ist ein UNIX-Betriebssystem mit offenem Standard, das auf verschiedenen Architekturen wie X86 (32-Bit und 64-Bit), Arm, Aarch64, RISC-V, MIPS, Powerpc und Sun Ultrassparpc läuft. Es bietet viele Funktionen wie Multitasking, Speicherschutz, virtuellen Speicher, Versprechen, SMP und Desktop mit X Windows, KDE und GNOME.

Funktionen von FreeBSD:

  • Kostenlose Open-Source-Lizenz: Der Quellcode kann ohne Copyleft-Einschränkungen frei geändert und erweitert werden.
  • Zuverlässige TCP/IP-Netzwerke: FreeBSD implementiert gängige und branchenübliche Protokolle sowohl für Server- als auch für Routing-/Firewall-Aufgaben.
  • Vollständig integrierte OpenZFS-Unterstützung: einschließlich Root-on-ZFS, ZFS usw.
  • Erweiterte Sicherheitsfunktionen, von der obligatorischen Zugriffskontrolle (MAC) bis hin zu Capsicum- und Sandboxing-Mechanismen.

Es steht eine breite Palette an Software zur Verfügung, darunter über 30.000 vorkonfigurierte Pakete und verschiedene Ports für die Erstellung individueller Software. FreeBSD basiert auf dem CSRG UC Berkeley 4.4BSD LiteRelease. Es gibt viele Anwendungen, für die es verwendet werden kann, darunter Desktop-Computing, Webdienste, Bildung, Forschung, Netzwerke, Gebäudesysteme und Softwareentwicklung.

Installation des Apache Webservers

  • Apache Webserver installieren

Für die Installation des Apache Webservers wird folgender Befehl aufgeführt:

pkg install apache24

Der Paketmanager wird nun den Apache Webserver und alle erforderlichen Abhängigkeiten herunterladen und installieren.

  • Automatischer Start des Apache-Dienstes beim Systemstart:

Um den Apache-Dienst automatisch beim Booten des Systems zu starten, wird die folgende Zeile zur Datei /etc/rc.conf hinzugefügt:

apache24_enable=YES

Diese Konfiguration bewirkt, dass der Apache-Dienst beim nächsten Systemstart automatisch gestartet wird.

  • Apache Webserver starten

Nach der Installation sollte der Apache Webserver automatisch gestartet werden. Falls nicht, kann dieser manuell gestartet werden:

  • Um den Apache Webserver zu starten, wird folgender Befehl ausgeführt:
service apache24 start

Der Apache-Dienst wird gestartet und ist nun einsatzbereit, um HTTP-Anfragen von Clients zu verarbeiten.

  • Stoppen des Apache-Dienstes:

Falls der Apache-Dienst gestoppt werden soll, wird der Befehl verwendet:

service apache24 stop

Der Apache-Dienst wird beendet und nimmt keine weiteren Anfragen mehr entgegen.

  • Neustarten des Apache-Dienstes:

Wenn Änderungen an der Konfiguration vorgenommen wurden oder der Apache-Dienst aktualisiert werden soll, kann dieser mit folgendem Befehl neu gestartet werden:

service apache24 restart

Dadurch wird der Apache-Dienst gestoppt und anschließend automatisch neu gestartet.

  • Status des Apache-Dienstes überprüfen:

Um den aktuellen Status des Apache-Dienstes anzuzeigen, wird folgender Befehl verwendet:

service apache24 status

Der Befehl zeigt den aktuellen Status des Dienstes an, ob er läuft oder gestoppt ist.

service: Dieser FreeBSD-Befehl wird zur Steuerung von Systemdiensten verwendet. Es ermöglicht es, Systemdienste zu starten, anzuhalten, fortzusetzen oder deren Status zu überprüfen.

apache24: Dies ist der Name des Dienstes für den Apache Webserver auf FreeBSD. Der Dienstename ist auf FreeBSD spezifisch und kann von dem auf anderen Betriebssystemen abweichen.

  • Konfiguration des Apache Webserver

Die primäre Apache-HTTP-Server-Konfigurationsdatei heißt httpd. conf, die sich normalerweise auf FreeBSD unter “/usr/local/etc/apache24/httpd.conf” befindet. Mehrere Einstellungen in dieser Datei regeln das Verhalten des Apache-Servers.

Im Folgenden sind die wichtigsten Abschnitte und Konfigurationsoptionen aufgeführt, die in der Datei httpd.conf zu finden sind:

  1. ServerRoot: Definieren Sie den Installationspfad für den Apache-Server.
  2. ServerName: Der vollständig qualifizierte Domänenname des Servers. Es wird verwendet, um URLs relativ zu sich selbst und zu Einstellungen auf Serverebene zu generieren.
  3. Listen: Der Port oder die IP-Adresse, die der Server abhört. Dies sind in der Regel Port 80 (HTTP) und 443 (HTTPS).
  4. LoadModule: Hier werden Apache-Module aktiviert, die zusätzliche Funktionalität bereitstellen. Beispielsweise ermöglicht LoadModule rewrite_module module/mod_rewrite.so das Neuschreiben von Modulen.
  5. DocumentRoot: Ein Verzeichnis ist der Verzeichniseinstiegspunkt einer Website oder Webanwendung.
  6. Directory: In diesem Abschnitt kann der DocumentRoot und die Ordneroptionen anderer Ordner konfiguriert werden. Dazu gehören Einstellungen wie Berechtigungen, Zugriffskontrolle und andere ordnerbezogene Einstellungen.
  7. VirtualHost: Hier wird die virtuelle Hostkonfiguration definiert, um mehrere Websites auf einem einzigen Server zu hosten. Jeder virtuelle Host kann seinen eigenen DocumentRoot, Hostnamen, Port und andere Einstellungen haben.
  8. ErrorLog: Dies ist der Pfad zu der Datei, in der Apache die Fehlermeldung protokolliert.
  9. CustomLog: Dies ist der Pfad zu der Datei, in der Apache Zugriffsprotokolle speichert.
  10. ServerSignature: Gibt an, ob die Serversignatur in der Serverantwort angezeigt werden soll. Es kann auf „Ein“ eingestellt werden. oder ausschalten".
  11. DirectoryIndex: Gibt die Standarddatei an, die der Server bereitstellen soll, wenn in der URL kein Dateiname angegeben ist (z.
    B.: index.html, index.php).
  12. MIME-Typen: In diesem Fall wird die Dateierweiterung mit den entsprechenden MIME-Typen verknüpft, um dem Browser mitzuteilen, wie er bestimmte Dateitypen interpretieren soll.
  13. AllowOverride: Ermöglicht es zu steuern, welche Einstellungen in der .htaccess-Datei im Verzeichnis überschrieben werden können. Dies gilt insbesondere bei der Verwendung verzeichnisspezifischer Konfigurationen.
  14. SSL-Konfiguration: Wenn der Apache-Server HTTPS unterstützt, enthält die Datei httpd.conf auch SSL/TLS-Zertifikatseinstellungen und Verschlüsselungskonfiguration.
  • Testen des Apache Webserver

Der Apache Webserver sollte nun einsatzbereit sein. Anschließend kann im Webbrowser "http://localhost/" eingegeben werden. Dadurch wird standardmäßig die Homepage des Apache-Webservers angezeigt.

Erklärung der rc.conf-Datei

Die rc.conf ist eine wichtige Konfigurationsdatei auf FreeBSD-Systemen, mit der verschiedenen Optionen und Konfigurationen angegeben werden können, die sich auf den Betrieb des Betriebssystems auswirken. Die Datei rc.conf enthält beschreibende Informationen zum lokalen Hostnamen, Konfigurationsdetails potenzieller Netzwerkschnittstellen und die Dienste, die beim ersten Systemstart ausgeführt werden. Bei Neuinstallationen wird die Datei rc.conf normalerweise vom Systeminstallationsprogramm ausgeführt. Der Zweck von rc.conf besteht nicht darin, Befehle auszuführen oder Aktionen direkt beim Systemstart auszuführen. Stattdessen wird es von verschiedenen gängigen Init-Skripten zu /etc hinzugefügt, wodurch deren Innenfunktionen von den dort gefundenen Einstellungen abhängig sind.
Sie befindet sich normalerweise im Verzeichnis /etc und wird von Benutzern mit Administratorrechten bearbeitet, um das System an ihre spezifischen Anforderungen anzupassen.

Die rc.conf beinhaltet:

- In der rc.conf-Datei
hostname="labpc-116"

Der Name des Systems im Netzwerk wird durch seinen Hostnamen angegeben.Unter diesem Namen wird das System bei der Kommunikation mit anderen Geräten erkannt und identifiziert. In diesem Fall lautet der Hostname „labpc-116“.

- In der rc.conf-Datei
keymap="de.kbd

Die Keymap-Variable gibt die Keymap für das System an. „de.kbd“ steht für deutsches Tastaturlayout.

-In der rc.conf-Datei
ifconfig_em0="DHCP

Die Variable ifconfig_em0 definiert die Netzwerkkonfiguration für die em0-Netzwerkschnittstelle. In diesem Fall ist die Schnittstelle auf DHCP (Dynamic Host Configuration Protocol) eingestellt. Dies bedeutet, dass der DHCP-Server des Netzwerks automatisch die IP-Adresse, die Subnetzmaske und das Gateway für das Netzwerk erhält. Somit wird die IP-Adresse, die Subnetzmaske und das Gateway des Netzwerks automatisch vom DHCP-Server bezogen, der mit dem Netzwerk verbunden ist.

- In der rc.conf-Datei
ifconfig_em0_ipv6="inet6 accept_rtadv

Wie die vorherige Variable aktiviert diese Variable IPv6-Einstellungen für die em0-Schnittstelle. „inet6“ zeigt an, dass IPv6 aktiviert ist und „accept_rtadv“ zeigt an, dass das System Router-Advertisement-Pakete akzeptiert, um die automatische Konfiguration der IPv6-Adresse zu ermöglichen.

- In der rc.conf-Datei
sshd_enable="YES

Mit sshd_enable wird der SSH-Dienst aktiviert. „YES“ bedeutet, dass der SSH-Server beim Systemstart läuft. Mit SSH können Benutzer eine sichere und verschlüsselte Verbindung zum System herstellen, um Befehle auszuführen und Dateien aus der Ferne zu übertragen.

 - In der rc.conf-Datei
 dumpdev="AUTO"

Die Variable dumpdev steuert das Verhalten des Crash-Dump-Speichers. Bei einem Crash-Dump handelt es sich um eine Aufzeichnung des Systemspeichers, wenn ein Absturz oder Fehler auftritt. Wenn „AUTO“ ausgewählt ist, ist die automatische Erkennung aktiviert und das System kann automatisch Absturzdumps generieren. Bei der Einstellung „NO“ wird das Absturz-Rendering deaktiviert.

 - In der rc.conf-Datei
 apache24_enable="yes

Konkret bedeutet apache24_enable=“yes”, dass der Apache HTTP-Server beim Systemstart automatisch startet. Wenn diese Option auf „YES“ gesetzt ist, wird der Apache-Dienst bei jedem Systemstart automatisch aktiviert und gestartet.

Beschreibung des Aufsetzens des Webservers unter den Supervise Tools

Im Rahmen des Projekts wurde die Entscheidung getroffen, Runit von Gerrit Pape als Lösung für das Daemon-Management unter FreeBSD zu nutzen. Runit stellt eine leistungsfähige Alternative zu herkömmlichen Init-Systemen dar und bietet interessante Funktionen.
Im Gegensatz zu herkömmlichen Techniken, bei denen ein Daemon normalerweise durch eine einzelne Datei dargestellt wird, verwendet Runit ein Verzeichnis zur Darstellung des Daemons. Dadurch können Konfigurationsdateien und Verwaltungstools zentral gespeichert werden, was die Verwaltung und Wartung des Daemons vereinfacht.
Der große Vorteil von Runit besteht darin, dass es nicht nur als Daemon-Manager fungiert, sondern auch als Ersatz für den Initialisierungsprozess verwendet werden kann. Dadurch wird das Systemstart- und -abschaltverhalten optimiert und verbessert.
Runit verfolgt den gleichen Ansatz wie Daemontools von Dan Bernstein und S6 von Laurent Berko, wobei die verknüpften Prozesse asynchron sind. Dadurch laufen Prozesse unabhängig voneinander ab und das Starten oder Stoppen eines Prozesses hat keine Auswirkungen auf andere Prozesse. Dies verbessern die Stabilität und Zuverlässigkeit des Systems.
Die Verwendung von Runit bietet eine leistungsstarke und flexible Methode zur Verwaltung von Daemons und ermöglicht eine effiziente Verwaltung verschiedener Dienste. Durch die klare Trennung der Dienstkonfigurationen in separate Ordner entsteht eine saubere und gut organisierte Umgebung.
Abschließend hat sich die Entscheidung, Runit in diesem Projekt einzusetzen, als äußerst vorteilhaft erwiesen. Mit Runit wurde eine solide und gut organisierte Infrastruktur für Daemons geschaffen, um eine reibungslose Verwaltung und einen zuverlässigen Betrieb der Dienste zu gewährleisten.

Installation und Konfiguration von Runit auf FreeBSD für effizientes Daemon-Management

Im Folgenden wird eine ausführliche Anleitung zur Konfiguration von Runit auf einem FreeBSD-System präsentiert:

  • Schritt 1: Erstellen eines Verzeichnisses und Berechtigungen setzen

Zunächst wird ein Verzeichnis mit dem Namen “package” im Stammverzeichnis (/) von FreeBSD erstellt. Es dient der Aufbewahrung von heruntergeladenen Paketen und Software.

mkdir -p /package
chmod 1755 /package

Die Erstellung des Verzeichnisses “/package” erfolgt mittels des Befehls “mkdir -p”. Die Option “-p” sorgt dafür, dass das Verzeichnis erstellt wird, auch wenn es bereits existiert. Anschließend werden die Berechtigungen des Verzeichnisses mit “chmod 1755” gesetzt. Dadurch wird sichergestellt, dass alle Benutzer das Verzeichnis lesen und ausführen können, jedoch nur der Root-Benutzer oder der Eigentümer das Recht haben, es zu beschreiben.

  • Schritt 2: Herunterladen und Entpacken von Runit

In dem vorliegenden Experiment wird der Webbrowser “lynx” im Terminal genutzt, um die Website "http://smarden.org" aufzurufen und die Runit-Version 2.1.2 von dieser Webseite herunterzuladen.

lynx http://smarden.org

Nach dem Herunterladen der Datei “runit-2.1.2.tar.gz” wird sie im Verzeichnis “/package” gespeichert.

cd /package
gunzip runit-2.1.2.tar.gz
tar -xpf runit-2.1.2.tar

Mittels “gunzip” wird die Datei “runit-2.1.2.tar.gz” entkomprimiert und anschließend wird “tar -xpf” verwendet, um die Tar-Datei zu entpacken und die ursprüngliche Verzeichnisstruktur wiederherzustellen.

  • Schritt 3: Löschen alter Dateien

Nachdem die Datei erfolgreich entpackt wurde, wird die ursprüngliche komprimierte Datei gelöscht, um Platz zu sparen.

rm runit-2.1.2.tar

Durch “rm” entfernen wir die tar-Datei, da sie nicht mehr benötigt wird.

  • Schritt 4: Vorbereitung für die Installation

Im Verzeichnis “/package/admin/runit-2.1.2” befindet sich ein automatisch erstelltes Verzeichnis mit dem Namen “admin”.

cd /package/admin/runit-2.1.2

Die Ordnerstruktur wird so erstellt, um mögliche Konflikte mit anderen Paket-Installationen zu vermeiden und die Verwaltung zu vereinfachen.

  • Schritt 5: Installation von Runit

Zur Installation von Runit auf FreeBSD führen wir den Befehl “package/install” im Verzeichnis “/package/admin/runit-2.1.2” aus.

cd /package/admin/runit-2.1.2
package/install

Das Verzeichnis “/package/admin/runit-2.1.2” enthält das Skript “install”, das ausgeführt wird, um Runit auf dem System zu installieren. Durch Ausführen dieses Skripts werden alle erforderlichen Schritte durchgeführt, um Runit zu kompilieren und zu installieren, einschließlich der Konfiguration von Runit für das FreeBSD-System.

  • Schritt 6: Anpassen und Kompilieren von Runit

In dem Verzeichnis “/package/admin/runit-2.1.2/src” sind die Konfigurationsdateien “conf-cc” und “conf-ld” zu finden. Dort wird der Eintrag “gcc” in beiden Dateien zu “cc” geändert, um den C-Compiler zu verwenden, der auf FreeBSD verfügbar ist.

cd /package/admin/runit-2.1.2/src
vi conf-cc
vi conf-ld

Das “make”-Tool, das für den Kompilierungsprozess benötigt wird, wird durch den Befehl “pkg install gmake” installiert:

sudo pkg install gmake

Nachdem das “make”-Tool erfolgreich installiert wurde, die Installation von Runit kann fortgesetzt werden. Um Runit auf einem FreeBSD-System zu installieren, navigiert man zum Verzeichnis “/package/admin/runit-2.1.2/src”, wo sich die Quellcode-Dateien von Runit befinden. Anschließend wird der Befehl “make” ausgeführt, um den Kompilierungsprozess zu starten und Runit zu erstellen:

cd /package/admin/runit-2.1.2/src
make

Die Nutzung von “make” ermöglicht es uns, Runit direkt aus dem Quellcode zu kompilieren und somit eine maßgeschneiderte Installation für unser System durchzuführen.

Anpassung der rc.local-Datei für Runit:

Nachdem Runit erfolgreich installiert und die “run”-Dateien für die Dienste erstellt wurden, muss die rc.local-Datei angepasst werden, um Runit beim Systemstart zu starten und die Dienste in /var/service zu überwachen. Hier sind die Anpassung und Erklärung:

Öffnen der rc.local-Datei:
Es wird ein Texteditor verwendet, um die rc.local-Datei zu öffnen und die erforderlichen Anpassungen vorzunehmen: cd vi /etc/rc.local

  • Anpassung des PATHs:

Zu Beginn der rc.local-Datei wird eine Zeile hinzugefügt, um den Systempfad (PATH) anzupassen und das Verzeichnis /usr/local/bin hinzuzufügen. Dadurch wird sichergestellt, dass Runit und seine zugehörigen Binärdateien gefunden werden können:

- In der rc.local-Datei
PATH=/usr/local/bin:$PATH
  • Starten von Runit und Diensten:

Nachdem der PATH angepasst wurde, wird folgender Befehl hinzugefügt, um Runit zu starten und die Dienste in /var/service zu überwachen:

- In der rc.local-Datei
csh -cf "/command/runsvdir /var/service &"

Erklärung: Der Befehl csh -cf wird verwendet, um einen C-Shell-Interpreter aufzurufen und das angegebene Kommando in demselben Kontext auszuführen. /command/runsvdir ist der Pfad zur Runit-Binärdatei “runsvdir”, die für das Verwalten und Überwachen der Dienste in /var/service verantwortlich ist. Wenn ein solcher Dienst gefunden wird, ruft “runsvdir” das entsprechende “runsv”-Kommando auf, das den Prozess-Supervisor “runsv” startet und den Dienst überwacht. Auf diese Weise arbeiten “runsvdir” und “runsv” zusammen, um eine effiziente Verwaltung und Überwachung aller Runit-Dienste in /var/service zu gewährleisten. Das “&” am Ende des Befehls bewirkt, dass das Kommando im Hintergrund ausgeführt wird, um den Systemstart nicht zu blockieren.

  • Speichern der Änderungen:

Nachdem die Anpassungen vorgenommen wurden, wird die rc.local-Datei gespeichert und der Texteditor wird geschlossen.

  • Reboot des Systems:

Nachdem die rc.local-Datei erfolgreich angepasst wurde, wird das System neu gestartet, damit Runit und die von ihm verwalteten Dienste beim Hochfahren des Systems automatisch gestartet werden.

  • Wichtigkeit der Anpassung der rc.local-Datei:

Die Anpassung der rc.local-Datei ist von entscheidender Bedeutung, um Runit beim Systemstart zu aktivieren und die Überwachung unserer Dienste in /var/service zu ermöglichen. Durch diese Anpassungen wird gewährleistet, dass die Dienste bereits vor dem Login-Prozess gestartet und überwacht werden, was die Verfügbarkeit und Stabilität unseres Servers oder Systems verbessert. Die korrekte Konfiguration der rc.local-Datei in Verbindung mit den “run”-Dateien von Runit bietet eine effiziente und zuverlässige Lösung für das Daemon-Management auf dem FreeBSD-System.

Anpassung der rc.conf-Datei für Runit:

Nach der erfolgreichen Installation und Konfiguration von Runit als Init-System und Prozess-Supervisor übernimmt Runit die Verwaltung der Dienste, einschließlich des Apache HTTP Servers. In diesem Fall ist die vorherige Einstellung apache24_enable=“yes” in der rc.conf nicht mehr erforderlich und kann auskommentiert oder entfernt werden. Dadurch entfällt die Notwendigkeit, spezifische Dienste in der rc.conf zu aktivieren, da Runit diese Funktionen übernimmt und die Dienste zuverlässig verwaltet.

Erstellung der “run”-Datei und ihre Bedeutung:

Nach der erfolgreichen Installation von Runit auf dem FreeBSD-System wird für jeden Dienst, der mit Runit verwaltet werden soll, eine “run”-Datei erstellt. Diese Datei ist entscheidend, da sie die Anweisungen enthält, die Runit ausführt, um den entsprechenden Dienst zu starten und zu verwalten.
Die Verzeichnisstruktur für die Dienste sieht wie folgt aus: Nach der Installation von Runit wird das Verzeichnis “/var/service/apache24” erstellt, um die Runit-Dienste zu speichern. Jeder Unterordner in diesem Verzeichnis repräsentiert einen Dienst, der von Runit verwaltet wird. Um einen Dienst zu konfigurieren, wird ein Verzeichnis mit dem entsprechenden Dienstnamen innerhalb von “/var/service/apache24” erstellt.
In jedem Dienstverzeichnis wird eine Datei namens “run” erstellt. Diese “run”-Datei enthält die Befehle, die von Runit ausgeführt werden, um den Dienst zu starten und zu überwachen. Jede “run”-Datei ist spezifisch für den zugehörigen Dienst und enthält Anweisungen, wie der Dienst gestartet, beendet und überwacht werden soll.

Die “run”-Datei ist ein Shell-Skript, das den Apache Webserver mithilfe von Daemontools und chpst (Change Process State) startet und verwaltet. Der Inhalt der “run”-Datei sieht folgendermaßen aus:

#!/bin/sh

Dies ist die sogenannte “Shebang”-Zeile und gibt an, dass das Skript mit der Shell /bin/sh ausgeführt werden soll.

exec 2>&1

Mit dieser Zeile wird die Standardfehlerausgabe (STDERR) des Skripts auf die Standardausgabe (STDOUT) umgeleitet. Dadurch werden Fehlermeldungen und Ausgaben, die normalerweise auf STDERR erscheinen würden, zusammen mit den normalen Ausgaben auf STDOUT angezeigt.

exec env -i /usr/local/bin/chpst -P

Hier wird das Programm chpst mit bestimmten Parametern und dem Apache Webserver (/usr/local/sbin/httpd) ausgeführt. Chpst wird verwendet, um den Webserver in einer kontrollierten Umgebung zu starten. Die Bedeutung der Parameter ist wie folgt:

  • P: Setzt die Prozessgruppe des Webserver-Prozesses, was für die Prozesskontrolle durch Daemontools wichtig ist und gibt, an, dass der Prent Prozess des ausführbarenden Programms nicht geändert werden soll.
  • /usr/local/sbin/httpd: Der Pfad zum Apache Webserver-Programm.
  • -f /usr/local/etc/apache24/httpd.conf: Gibt die Konfigurationsdatei httpd.conf an, die vom Webserver verwendet werden soll.
  • -DFOREGROUND: Startet den Webserver im Vordergrund und hält das Skript offen, solange der Webserver läuft.
2>&1

Dies ist erneut eine Umleitung, die sicherstellt, dass sowohl die Standardausgabe als auch die Standardfehlerausgabe in der gleichen Datei landen, nämlich der Datei, die in Daemontools verwendet wird, um die Ausgabe des laufenden Prozesses zu protokollieren.

  • Starten und Überwachen des Dienstes:

Nachdem die “run”-Datei erstellt wurde, übernimmt Runit die Kontrolle über den Dienst. Mithilfe der Informationen aus der “run”-Datei startet Runit den Dienst und überwacht ihn kontinuierlich, um sicherzustellen, dass er ordnungsgemäß ausgeführt wird. Falls der Dienst aus irgendeinem Grund beendet wird, wird Runit ihn automatisch neu starten, um die Verfügbarkeit und Stabilität des Dienstes zu gewährleisten. Dieses automatische Neustartverhalten von Runit trägt dazu bei, dass der Dienst zuverlässig und ohne längere Ausfallzeiten zur Verfügung steht. Dies ist besonders wichtig für eine reibungslose Funktionsweise des Systems und für eine effiziente Verwaltung der Dienste auf dem FreeBSD-System.

  • Manuelles Starten und Überwachen eines Runit-Dienstes mit sv-Befehlen:

Die folgende Anleitung beschreibt die grundlegenden sv-Befehle, mit den einen Runit-Dienst manuell gestartet, überwacht, gestoppt und neu gestartet werden kann. Diese Befehle ermöglichen es, die Kontrolle über die Dienste auf dem FreeBSD-System zu haben und sicherzustellen, dass sie ordnungsgemäß funktionieren.

  1. Dienst manuell starten

Um einen Runit-Dienst manuell zu starten, wird der Befehl sv start <dienstname> verwendet:

sv start /var/service/apache24
  1. Status des Dienstes überprüfen

Um den Status eines Dienstes zu überprüfen, wird der Befehl sv status <dienstname> verwendet:

sv status /var/service/apache24

Der Befehl zeigt den aktuellen Status des Dienstes an, ob er läuft oder gestoppt ist.

  1. Dienst stoppen

Um einen Dienst zu stoppen, wird der Befehl sv stop <dienstname> verwendet:

sv stop /var/service/apache24
  1. Dienst neu starten

SSL/TLS-Zertifikate - Bedeutung und Zweck:

Die Datenübertragung im Internet wird durch digitale Zertifikate, sogenannte SSL/TLS-Zertifikate, gesichert und verschlüsselt. Zweck und Bedeutung von SSL/TLS-Zertifikaten werden im Folgenden erläutert:

  • Datenverschlüsselung:

Der Hauptzweck eines SSL/TLS-Zertifikats besteht darin, die Kommunikation zwischen dem Webbrowser des Benutzers und dem Webserver zu verschlüsseln. Durch die Verwendung von SSL/TLS wird sichergestellt, dass vertrauliche Informationen wie Benutzernamen, Passwörter, Kreditkarteninformationen und persönliche Daten während der Übertragung vor unbefugtem Zugriff geschützt sind. Die Daten werden verschlüsselt übertragen und können nur von autorisiertem Personal entschlüsselt werden, wodurch das Risiko, dass Daten von Dritten gelesen und abgefangen werden, erheblich reduziert wird.

  • Authentifizierung der Serveridentität:

Mit SSL/TLS-Zertifikaten kann ein Webserver authentifiziert und überprüft werden, ob ein Benutzer tatsächlich mit der beabsichtigten Website oder dem beabsichtigten Dienst interagiert. Zertifikate werden von vertrauenswürdigen Zertifizierungsstellen (Certification Authorities-CAs) ausgestellt, um die Identität des Zertifikatsinhabers zu überprüfen.Wenn ein Webserver ein gültiges SSL/TLS-Zertifikat besitzt, kann der Webbrowser die Authentizität des Servers überprüfen und sicherstellen, dass die Verbindung sicher und vertrauenswürdig ist.

  • Vertrauenswürdige Kommunikation:

Um das Vertrauen der Benutzer in Websites zu erhöhen, sind SSL/TLS-Zertifikate von entscheidender Bedeutung. Die meisten Betriebssysteme und Webbrowser akzeptieren Zertifikate, die von vertrauenswürdigen Zertifizierungsstellen ausgestellt wurden. Dieses visuelle Feedback stärkt das Vertrauen der Benutzer, indem es versichert, dass die Informationen sicher sind und dass sie eine legitime und sichere Website anzeigen.

  • SEO-Vorteile:

Suchmaschinen wie Google bevorzugen Websites, die HTTPS nutzen und nutzen HTTPS als Rankingfaktor. Dadurch können Seiten, die SSL/TLS-Zertifikate verwenden und auf HTTPS umstellen, in den Suchergebnissen besser ranken und eine größere Sichtbarkeit und Reichweite erzielen.

umbenannt. Diese werden zur Generierung von SSL/TLS-Zertifikaten für den Server verwendet.

chmod +x make_server.sh

ausgeführt, um das Skript ausführbar zu machen.

Anschließend wird das Skript “make_server.sh” mit dem Befehl:

gestartet. Dadurch werden aufgefordert, zusätzliche Informationen anzugeben, die zum Generieren eines Root Certificate Authority (CA)-Zertifikats erforderlich sind. Nach der Eingabe aller erforderlichen Informationen kann das Skript ein gültiges X.509-Zertifikat für die CA generieren.

Dieser Vorgang generiert ein gültiges X.509-Zertifikat für den „labpc-116“-Server. Mit diesem Zertifikat sind sowohl Authentifizierung als auch die sichere Netzwerkkommunikation möglich. Dadurch wird die Sicherheit und Zuverlässigkeit der Kommunikation mit den Servern verbessert.

Nachdem das Zertifikat für die Root-Zertifizierungsstelle (CA) erfolgreich erstellt wurde, dessen Details und Eigenschaften können angezeigt werden. Dazu wird der folgende Befehl verwendet:

openssl x509 -in labpc-116_ECCert.pem -text -noout | more

Dieser Befehl nutzt das OpenSSL-Werkzeug, um das X.509-Zertifikat mit dem Dateinamen “labpc-116_ECCert.pem” im Textformat zu entschlüsseln und anzuzeigen. Dabei werden wichtige Informationen wie der Aussteller, der Inhaber und der Gültigkeitszeitraum des Zertifikats dargestellt. Die Ausgabe könnte möglicherweise sehr umfangreich sein, daher wir der Befehl “more” verwendet, um die Anzeige seitenweise zu ermöglichen und somit eine bessere Übersicht zu erhalten.

Konfiguration der httpd.conf-Konfigurationsdatei:

Die “httpd.conf”-Datei ist die Hauptkonfigurationsdatei des Apache Webserver und enthält wichtige Einstellungen und Direktiven, die das Verhalten des Servers steuern. Hier ist eine ausführliche Erklärung der einzelnen Abschnitte und Direktiven der “httpd.conf”:

  • ServerRoot:
ServerRoot "/usr/local"

Die ServerRoot-Direktive legt das Basisverzeichnis für den Server fest, in dem der Apache Webserver installiert ist. Hier wird “/usr/local” als Serverstammverzeichnis definiert.

  • ServerName und Listen:
ServerName localhost
Listen 80
Listen 443

Die ServerName-Direktive definiert den Standard-Servernamen, der verwendet wird, wenn keine spezifische Servername-Angabe in einem Virtual Host erfolgt. Hier ist der Servername als “localhost” festgelegt.

Die Listen-Direktive gibt an, auf welchen Ports der Server lauschen soll. In diesem Fall lauscht er auf Port 80 für HTTP und auf Port 443 für HTTPS (SSL/TLS).

  • User und Group:
User www
Group www

Die User- und Group-Direktiven legen den Benutzernamen und die Gruppe fest, unter denen der Apache-Prozess ausgeführt wird. Hier wird der Benutzer “www” und die Gruppe “www” für den Apache-Prozess verwendet.

  • DocumentRoot und DirectoryIndex:
DocumentRoot "/usr/local/www/apache24/data"
DirectoryIndex index.html

Die DocumentRoot-Direktive gibt das Standardverzeichnis an, in dem die Inhalte des Servers liegen. Hier wird “/usr/local/www/apache24/data” als das Hauptverzeichnis festgelegt.

Die DirectoryIndex-Direktive gibt die Standarddatei an, die der Server anzeigen soll, wenn keine spezifische Datei angefordert wurde. Hier ist “index.html” als Standarddatei festgelegt.

  • VirtualHost-Direktiven:

*VirtualHost :80:

<VirtualHost *:80>
    DocumentRoot "/usr/local/www/iphone"
    <Directory "/usr/local/www/iphone">
        Options FollowSymLinks
        AllowOverride AuthConfig
    </Directory>
    DirectoryIndex index.html
</VirtualHost>

Dieser virtuelle Host ist für Port 80 (HTTP) konfiguriert. Hierbei wird keine Verschlüsselung eingesetzt und die Datenübertragung zwischen Webbrowser und Webserver erfolgt im Klartext.Dies bedeutet, dass die übertragenen Daten unverschlüsselt sind und die Gefahr besteht, dass sie von Dritten abgefangen und gelesen werden.
Das DocumentRoot ist /usr/local/www/iphone und die verzeichnisspezifischen Optionen sind auf /usr/local/www/iphone eingestellt, einschließlich FollowSymLinks und AuthConfig.

*VirtualHost :443:

<VirtualHost *:443>
    DocumentRoot "/usr/local/www/iphone"
    <Directory "/usr/local/www/iphone">
        Options FollowSymLinks
        AllowOverride AuthConfig
    </Directory>
    DirectoryIndex index.html

    SSLCertificateFile "/root/labpc-116_ECCert.pem"
    SSLCertificateKeyFile "/root/labpc-116_ECCkey.pem"
</VirtualHost>

Dieser virtuelle Server ist auf Port 443 (HTTPS) konfiguriert und stellt eine verschlüsselte Verbindung über SSL/TLS bereit. Die Anweisungen SSLCertificateFile und SSLCertificateKeyFile geben den Pfad zum SSL/TLS-Zertifikat an, das für die Verschlüsselung verwendet wird. Mithilfe dieser Zertifikate verschlüsselt der Webserver die Kommunikation zwischen dem Webbrowser des Benutzers und dem Webserver und gewährleistet so die Sicherheit der übertragenen Daten.
Das DocumentRoot ist ebenfalls „/usr/local/www/iphone“ und legt die gleichen Verzeichnisoptionen wie im vorherigen virtuellen Host fest.
Außerdem wird der Pfad für das SSL/TLS-Zertifikat angegeben.

  • LogLevel, LogFormat, ErrorLog und CustomLog:
LogLevel debug
LogFormat "%h %l %u %t \"%r\" %>s %b \"%{Referer}i\" \"%{User-Agent}i\"" common
ErrorLog /var/log/apache/error.log
CustomLog /var/log/apache/access.log common

Die LogLevel-Direktive legt das Protokollierungsniveau für den Server fest. Hier wird “debug” als Protokollierungsniveau verwendet.

Die LogFormat-Direktive definiert das Format für die Zugriffsprotokolle. Hier wird das Format “common” festgelegt.

Die ErrorLog-Direktive gibt den Pfad zur Datei an, in der Apache Fehlerprotokolle protokolliert.

Die CustomLog-Direktive gibt den Pfad zur Datei an, in der Zugriffsprotokolle für den Server protokolliert.

  • LoadModule:
LoadModule mpm_event_module libexec/apache24/mod_mpm_event.so
LoadModule log_config_module libexec/apache24/mod_log_config.so
LoadModule mime_module libexec/apache24/mod_mime.so
LoadModule authz_core_module libexec/apache24/mod_authz_core.so
LoadModule dir_module libexec/apache24/mod_dir-so
LoadModule unixd_module libexec/apache24/mod_unix.so
LoadModule ssl_module libexec/apache24/mod_ssl.so

Sie werden verwendet, um Apache-Module zu laden, die bestimmte Funktionen und Features bereitstellen, die der Webserver benötigt, um verschiedene Aufgaben auszuführen.

  1. mpm_event_module:

Dieses mpm_event_module stellt das Multiprocessing Module (MPM)-Ereignis bereit. Das MPM-Ereignis ermöglicht es dem Apache-Webserver, mehrere Anfragen gleichzeitig in separaten Threads zu verarbeiten. Es bietet eine gute Leistung und Skalierbarkeit, insbesondere für stark ausgelastete Websites.

  1. log_config_module:

Mit dem log_config_module kann die Protokollierungseinstellungen des Apache-Webservers konfiguriert werden. Das Format des Zugriffsprotokolls kann angepasst werden, um verschiedene Informationen über Benutzeranfragen und -aktivitäten zu sammeln.

  1. mime_module:

Das mime_module ist für die Verarbeitung von MIME-Typen (Multipurpose Internet Mail Extensions) verantwortlich. Dadurch kann der Server den richtigen MIME-Typ für Dateien ermitteln und senden, was für Webbrowser wichtig ist, um Dateien richtig zu interpretieren.

  1. authz_core_module:

Das authz_core_module stellt grundlegende Autorisierungs- und Zugriffskontrollfunktionen für den Apache-Webserver bereit. Dadurch können die Zugriffsrichtlinien basierend auf Benutzerberechtigungen und anderen Kriterien konfiguriert werden.

  1. dir_module:

Mit dem dir-module können automatisch Verzeichnislisten erstellen werden, wenn im Verzeichnis keine Indexdatei vorhanden ist. Außerdem können die Einstellungen für bestimmte Ordner angepasst werden.

  1. unixd_module:

Das unixd-module stellt viele der Unix-spezifischen Funktionen bereit, die zum Ausführen des Apache-Webservers auf Unix-basierten Betriebssystemen erforderlich sind.

  1. ssl_module:

Das SSL-module stellt die für SSL/TLS-Verschlüsselung und HTTPS-Verbindungen erforderliche Funktionalität bereit. Dadurch kann der Server SSL/TLS-Zertifikate verwenden, um die Kommunikation zu verschlüsseln und die Sicherheit zwischen dem Server und dem Client zu gewährleisten.

Durch das Laden dieser Module kann der Apache Webserver eine Vielzahl von Funktionen ausführen und eine breite Palette von Webanwendungen unterstützen. Jedes Modul erweitert den Funktionsumfang des Servers und ermöglicht es, verschiedene Aspekte der Serverkonfiguration anzupassen und den Webserver gemäß Ihren Anforderungen und Bedürfnissen zu konfigurieren. Es ist wichtig, nur die benötigten Module zu laden, um die Leistung des Webservers zu optimieren und unnötige Ressourcenverschwendung zu vermeiden.

  • TypesConfig:

Die TypesConfig-Direktive gibt den Pfad zur Datei mit den MIME-Typen an.

Nachdem die httpd.conf Konfigurationsdatei angepasst wurde, werden die Befehle lynx https://localhost und lynx localhost:80 ausgeführt, um die Funktionalität und Erreichbarkeit des Apache Webserver zu überprüfen.

lynx https://localhost

Mit diesem Befehl wird der Text-Browser lynx verwendet, um die lokale Adresse https://localhost aufzurufen. Das bedeutet, es wird eine verschlüsselte HTTPS-Verbindung zum Webserver aufgebaut. Da in der httpd.conf ein Virtual Host für den Port 443 (HTTPS) konfiguriert wurde und die entsprechenden SSL/TLS-Zertifikate angegeben wurden, sollte lynx eine erfolgreiche Verbindung zu der Website aufbauen und den Inhalt der Seite anzeigen. Wenn dies gelingt, bestätigt es, dass die SSL/TLS-Verschlüsselung ordnungsgemäß funktioniert und der Webserver auf Anfragen über HTTPS reagiert.

lynx localhost:80

Mit diesem Befehl wird der Text-Browser lynx verwendet, um die lokale Adresse localhost:80 aufzurufen. Dieser Aufruf erfolgt über den Standard-HTTP-Port 80, was bedeutet, dass keine Verschlüsselung verwendet wird. Der Webserver sollte auf die Anfrage reagieren und den Inhalt der Seite anzeigen. Wenn dies gelingt, bestätigt es, dass der Webserver auch auf HTTP-Anfragen über den Port 80 reagiert und Websites ohne Verschlüsselung bereitstellt.

Konfiguration der Logdateien-Rotation in der newsyslog.conf-Datei

Der newsyslog Dienst ist dafür verantwortlich, Logdateien zu rotieren, indem er ältere Logdateien archiviert oder löscht und neue Logdateien erstellt. Dadurch wird verhindert, dass Logdateien zu groß werden und unnötig viel Speicherplatz belegen.
Hier eine allgemeine Erläuterung der Funktion der Datei newsyslog.conf:

  1. Rotationsregeln definieren: Die Datei newsyslog.conf definiert Rotationsregeln für verschiedene Logdateien. Jede Regel gibt an, welche Logdateien wie oft rotiert werden sollen, wie viele Archive erstellt werden sollen und welche Berechtigungen neue Logdateien haben sollten.
  2. Konfiguration der Logdatei: Die Datei newsyslog.conf enthält Zeilen, die den Pfad zur zu rotierenden Protokolldatei auflisten. Diese Zeilen enthalten auch zusätzliche Informationen wie gewünschte Rotationfrequenz, maximale Protokolldateigröße und andere Optionen.
  3. Archivdateiverwaltung: Wenn eine Logdatei geändert wird, erstellt newsyslog Archive, um alte Logeinträge zu speichern. Die Anzahl der Archive und deren Namen werden ebenfalls in der Datei newsyslog.conf angegeben.
  4. Benutzerdefinierte Einstellungen: Mit der Datei newsyslog.conf können Benutzer benutzerdefinierte Einstellungen für einige Logdateien festlegen, die von den Standardregeln abweichen können.

Die Datei newsyslog.conf enthält die Konfiguration zum Rotieren der Logdateien, auch hier sind die gleichen Pfade vorgesehen wie in der Datei httpd.conf. Diese Kopie wird benötigt, damit newsyslog weiß, welche Logdateien rotiert werden müssen.
Zur Rotation sehen die Einträge in der Datei newsyslog.conf wie folgt aus:

- In der newsyslog.conf Datei
/var/log/apache/access.log    644  7     *    @T00   JC    /var/run/httpd.pid 
/var/log/apache/error.log     644  7     *    @T00   JC   /var/run/httpd.pid

Hier ist die Bedeutung jedes Parameters:

  1. /var/log/apache/access.log (logfilename): Dies ist der Pfad zur Logdatei, die rotiert werden soll. In diesem Fall handelt es sich um die Zugriffsprotokolldatei des Apache-HTTP-Servers unter /var/log/apache/access.log.
  2. 644 (mode): Dieser Modus wird für neu erstellte Logdateien verwendet, nachdem sie rotiert wurden. In diesem Fall werden neue Logdateien mit der Berechtigung 644 erstellt (von Eigentümer und Gruppe gelesen, aber nicht ausgeführt).
  3. 7 (count): Dies ist ein Zählerparameter, der die Anzahl der alten Logdateien angibt, die nach der Rotation erhalten bleiben. In diesem Fall werden 7 alte Logdateien archiviert.
    • (size): Der Größenparameter gibt die maximale Größe der Logdatei an, bevor sie rotiert wird. Die Angabe * bedeutet, dass die Datei je nach Größe rotiert wird.
  5. @T00 (when): Der Parameter when gibt an, wann die Rotation stattfindet. In diesem Fall wird @T00 verwendet, was bedeutet, dass die Rotation jeden Tag um Mitternacht erfolgt.
  6. JC (flags): Der Flags-Parameter definiert, welche Aktion während der Rotation ausgeführt werden soll. Hier steht JC für „copytruncate“, was bedeutet, dass die Logdatei vor dem Rotieren kopiert und dann gelöscht wird, um Platz für neue Protokolleinträge zu schaffen.
  7. /var/run/httpd.pid (pid_file): Der Parameter enthält den Pfad zur PID-Datei des Apache HTTP Servers.
    Diese PID-Datei wird während der Rotation verwendet, um den Apache-Prozess zu identifizieren und die Registrierung fortzusetzen, ohne den Prozess neu zu starten.
